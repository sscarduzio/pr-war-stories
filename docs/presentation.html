The Problem

🧠

AI Reviewers Are Amnesiac

Cursor Bugbot, CodeRabbit, and Copilot review every PR from scratch. They don't know your team spent a week debugging a race condition last month, or that a certain === check is intentional.

"The syntax is correct" != "The approach is safe"

📝

Knowledge Dies on Merged PRs

A senior engineer writes: "Be careful, MemoryStorageAdapter preserves object references intentionally." That comment lives on a PR nobody reads again. Three months later, someone "fixes" the === check.

Same bug. Same review cycle. Same wasted time.

🔇

Context Overload Kills Quality

Dumping 50 rules into a single config file means every rule competes for the bot's attention. Important rules get drowned out by generic advice. The bot starts ignoring everything.

More rules != better reviews

🔓

Open Feedback Loop

Even when lessons are captured, there's no systematic way to feed them back into the review process. Knowledge stays in people's heads or in Slack threads nobody searches.

Institutional knowledge is the most perishable asset

The Solution: Three Memory Layers

Layer 1

.cursor/BUGBOT.md

Read by Bugbot at review time

Project-specific rules the bot enforces on every diff. Hierarchical: the bot traverses upward from each changed file, collecting all BUGBOT.md files it finds.

"Never use Promise.all on unbounded arrays. Use a concurrency limiter."
Layer 2

LESSONS.md

Read by IDE assistants at write time

Universal engineering lessons read by Claude Code and Cursor before developers write code. The "why" behind patterns that can't be checked on a diff.

"In-memory state is a cache, not a source of truth. Query the database for authoritative conflict detection."
Layer 3

Inline Comments

Visible in diff when file is changed

Rules that apply to exactly one file or function, placed in the source code. The bot sees them whenever that specific file appears in a PR.

// WARNING: Runtime-evaluated // JS string. Cannot use // imports. Do not DRY. // (See PR #760)
3 / 9

Hierarchical Scoping

Deeper modules get more context. Simple changes get only relevant rules.

.cursor/BUGBOT.md Every PR
apps/octostar/.cursor/BUGBOT.md Octostar PRs
apps/octostar/.../linkchart-nt/.cursor/BUGBOT.md LinkChart PRs
apps/search-app/.cursor/BUGBOT.md Search PRs
packages/.cursor/BUGBOT.md Package PRs

Rule Classification System

Reviewable

Bot can check it

→ BUGBOT.md

"Don't use Promise.all on unbounded arrays"

Educational

Informs devs

→ LESSONS.md

"Build order requires packages before apps"

Single-File

One function

→ inline comment

"This adapter uses reference equality intentionally"

Overlapping

Duplicate rule

→ merge

Three stale-state rules merged into one

Stale

Pattern fixed

→ remove

API migrated, old workaround no longer needed

⚠️

The #1 failure mode: dumping everything into BUGBOT.md. This classification prevents context bloat and ensures each rule is placed where it's most effective.
