Rule Classification System

Reviewable

Bot can check it

→ BUGBOT.md

"Don't use Promise.all on unbounded arrays"

Educational

Informs devs

→ LESSONS.md

"Build order requires packages before apps"

Single-File

One function

→ inline comment

"This adapter uses reference equality intentionally"

Overlapping

Duplicate rule

→ merge

Three stale-state rules merged into one

Stale

Pattern fixed

→ remove

API migrated, old workaround no longer needed

⚠️

The #1 failure mode: dumping everything into BUGBOT.md. This classification prevents context bloat and ensures each rule is placed where it's most effective.

📐

Architectural Knowledge

What was decided and why
Written at decision time
Forward-looking
Predictable consequences

Audience: future architects

vs
👻

Shadow Knowledge

What went wrong and what we learned
Emerges after the fact
Backward-looking
Unknowable at decision time
Lives in PR comments, people's heads
Audience: future developers
Example war story
"If you bypass the epoch check or call markDone() directly, pending counts go negative and the UI shows stale loading forever."
ADRs document the skeleton. Shadow knowledge is the muscle memory.
It only exists in the heads of people who've been burned. And it walks out the door when they leave.

The Skill Commands

Five commands, each for a different moment in the lifecycle

/pr-war-stories setup Once per repo
Mines 50+ merged PRs, creates hierarchical BUGBOT.md files, LESSONS.md, installs the harvest GitHub Action, wires CLAUDE.md. Full bootstrap.
harvest-lessons.yml Auto — every PR merge
GitHub Action fires automatically. Extracts substantive human review comments, posts a structured harvest summary on the merged PR. No human trigger needed.
/pr-war-stories harvest When harvest comments appear
Reads harvest summaries, classifies each lesson (reviewable / educational / single-file), places it in the right layer. The human-in-the-loop step.
/pr-war-stories recheck After big refactors
Greps every rule for path and function references, verifies they still exist in the codebase. Flags stale scopeRules prefixes. Reports but does not auto-fix.
/pr-war-stories audit Quarterly
Checks Bugbot hit rate on last 20 PRs. Removes rules that never triggered. Merges duplicates. Graduates rules that can now be linted. Prevents rot.

Questions you're probably thinking

Anticipated from senior engineers, architects, and tech leads

Won't BUGBOT.md rot like every other living doc?
The quarterly audit phase catches staleness: rules that never trigger get removed, fixed patterns get graduated. The harvest Action auto-surfaces new lessons, so fresh rules keep arriving while stale ones get pruned. Entropy is real — the audit cadence is the answer.
Why not just write better linter rules or tests?
If a rule can be linted or tested deterministically, it should be — and then removed from BUGBOT.md. BUGBOT rules are for fuzzy, contextual knowledge: “this === is intentional,” “these two query paths must stay in sync.” Things that need understanding intent, not pattern matching.
How do you keep the token budget from creeping?
<400 words per file, <2,000 tokens worst case. Classification is the enforcer: reviewable → BUGBOT, educational → LESSONS, single-file → inline comment. Rules that don't fit any bin don't get added. Fewer rules = more attention per rule.
What about cross-cutting rules that span modules?
Root .cursor/BUGBOT.md. The bot traverses upward, so every PR sees the root rules. Module-level files add specificity, not duplication. If the same bug recurs in a different module, promote the rule upward.
Does this actually work with Bugbot / CodeRabbit / Copilot?
Any reviewer that reads .cursor/BUGBOT.md from the repo (Cursor Bugbot does this natively). For others: LESSONS.md and inline comments are universal — they're just files in the repo that any AI assistant can be told to read.
What if a reviewer's suggestion wasn't adopted?
Rejected suggestions are not rules. The harvest step explicitly filters for this: only corrections that were actually adopted, and warnings about real pitfalls, become rules. Style preferences and debated alternatives stay as PR conversation.
How do you handle module renames and big refactors?
BUGBOT.md files move with git mv, but rules inside may reference stale paths. After any big refactor: 1) grep BUGBOT.md files for moved/deleted paths, 2) update scopeRules in harvest-lessons.yml, 3) check inline comments didn't lose context. LESSONS.md is path-agnostic — no impact. Add this to your refactor PR checklist.
Is the harvest Action noisy on active repos?
It only posts when substantive human comments exist (>40 chars, not LGTM/approvals, not bots). Most PRs with only bot reviews produce nothing. On a repo merging 5 PRs/day, expect 1–2 harvest summaries per week.

Post Scriptum — Future Research

AI That Sees and Uses Our UI Like a Human

Instead of writing brittle test scripts that break every time the UI changes, new open-source tools let us describe what we want in plain English — “click the submit button”, “fill in the form”, “drag this item to the list” — and an AI model looks at the screen and does it.

Automation SDK

Midscene.js

Open-source automation SDK that plugs into our existing Playwright setup. Understands UI elements visually — no selectors, no XPath, no data-testid.

Vision Model

UI-TARS

Open-source AI model trained specifically to interact with user interfaces. Sees screenshots, plans actions, executes multi-step workflows autonomously.

Both run entirely on our hardware. No cloud. No licensing costs. No data leaves our network.

Post Scriptum — Future Research

Promising, But Needs Validation

Early signals are strong. The gap is between demo-grade and production-grade.

What looks good

  • Works on our existing GPU (RTX 6000 PRO) with zero new infrastructure
  • Handles complex UIs including drag & drop, nested menus, dynamic content
  • Active open-source ecosystem, evolving fast — Midscene reached 1.0 in 2026
  • Natural-language test specs are readable by non-engineers — QA, product, design can write them
  • Tests survive UI refactors that break traditional selectors — the model re-finds elements visually

What we need to test

  • Speed — can it run a full regression suite in CI time budgets?
  • Determinism — do tests produce consistent results across runs?
  • Octostar-specific UIs — link charts, record viewers, complex tables
  • Failure diagnostics — when a test fails, can you understand why?
  • GPU contention — model inference alongside dev workloads on the same RTX 6000
