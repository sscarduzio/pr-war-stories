How it works

Three memory layers,
each optimized for a different reader

Not all knowledge belongs in the same place. Rules go where they're most effective.

Layer 1 -- Bot memory

.cursor/BUGBOT.md

Read by Bugbot at review time

Hierarchical rules the bot checks against every diff. It traverses upward from each changed file, collecting all BUGBOT.md files it finds. Token-budgeted: fewer rules = more attention per rule.

"Never use Promise.all on unbounded arrays. Use a concurrency limiter."
Layer 2 -- Developer memory

LESSONS.md

Read by IDE assistants at write time

Universal engineering lessons read by Claude Code and Cursor before developers write code. The "why" behind patterns -- things the bot can't enforce but developers should know.

"In-memory state is a cache, not a source of truth. Query the database for authoritative conflict detection."
Layer 3 -- File memory

Inline comments

Visible in diff when that file changes

Rules that apply to exactly one file or function, placed in the source code. The bot sees them whenever that specific file appears in a PR. Most targeted, zero waste.

// WARNING: Runtime-evaluated // JS string. Cannot use imports. // Do not DRY. (See PR #760)

Real examples

War stories from production

These are actual lessons extracted from merged PRs. Each one prevented the same class of bug from recurring.

From a production OOM

Unbounded Promise.all on file uploads causes OOM crashes

Hundreds of simultaneous mkdir + upload requests for deep folder trees exhausted memory. Fixed with concurrency limiter (max 3 parallel ops).

BUGBOT.md
From a reviewer warning

LLM-generated CSS changes broke unrelated components

"An example of how a large scale change by an LLM can cause unintended changes to other components." A sidebar CSS change collapsed the schema editor.

BUGBOT.md
From a false "fix"

MemoryStorageAdapter reference equality is intentional

Bot flagged === as a bug. Maintainer explained: the adapter preserves object references for unchanged items. Deep equality would trigger unnecessary re-renders.

Inline comment
From a timing bug

useState captures values one frame late

priorPodUid was in useState. The effect to capture it ran one frame late. Replaced with useRef for synchronous capture during state transitions.
